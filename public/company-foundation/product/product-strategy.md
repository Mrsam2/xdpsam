# Product Strategy

## How we decide what to build

We work backwards from **customer problems**, not from features or technology. A problem must be real, frequent, and painful enough that customers will pay to solve it.

## Product principles

- **Problem first, solution second.** Validate the problem before building.
- **Smallest valuable thing.** Ship the minimal slice that delivers real value, then iterate.
- **Outcomes over output.** Measure impact (adoption, retention, revenue), not features shipped.
- **Productize patterns.** When we solve the same problem repeatedly in consulting, consider turning it into a product.

## Discovery → Delivery

| Phase | Question | Output |
|---|---|---|
| Discover | Is this a real, valuable problem? | Validated problem, target user |
| Define | What's the smallest solution that works? | Scoped MVP, success metrics |
| Build | Can we deliver it with quality? | Working increment |
| Measure | Did it move the metric? | Data + learnings |
| Decide | Persevere, pivot, or kill? | Next bet |

## Prioritization

Use a simple, transparent framework (e.g., **RICE**: Reach × Impact × Confidence ÷ Effort) and keep a single prioritized backlog. Founders ratify the roadmap quarterly alongside OKRs.

## From services to SaaS (XDP's's path)

1. Deliver custom client work (cash + learning).
2. Spot repeated patterns and build internal accelerators.
3. Package into **productized services**.
4. Extract a **SaaS product** when the pattern generalizes across customers.
